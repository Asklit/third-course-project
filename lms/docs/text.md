На практике в текущем процессе есть несколько проблем:
1. Во первых преподаватели вынуждены вручную выгружать отчеты и код с lms
2. Во вторых ассистенты вынуждены ждать выгрузки работ преподавателями для проверки.
3. Для студентов поиск по материалам ограничен возможностями Word и не работает по всей базе лабораторных работ.

Из-за этого процесс текущий процесс обучения не удобен как для студентов, так и для преподавателей и ассистентов.

Объектом исследования является организация учебного процесса дисциплины «Программирование» НИУ ВШЭ – Пермь


Предметом исследования выступают алгоритмы для загрузки работ и предоставления доступа к базе данных учебных материалов

---

**Цель работы** — разработка сервиса, обеспечивающего загрузку результатов выполнения лабораторных работ и предоставляющего доступ к материалам курса с возможностью поиска по ним.  

**Основные задачи работы:**
1. Провести анализ предметной области, целевой аудитории и существующих решений.
2. Сформировать функциональные и нефункциональные требования.
3. Спроектировать архитектуру системы, базы данных и пользовательский интерфейс.
4. Реализовать инфраструктуру, ядро системы, микросервис Wiki и клиентскую часть.
5. Выполнить тестирование бизнес-логики микросервисов.

---
 
В качестве основного аналога для сдачи работ можно отметить **SMART LMS**, которая уже используется в вузе. Она позволяет студентам авторизоваться, просматривать статусы работ и сдавать файлы. 
Однако у нее есть важный недостаток: отсутствует автоматизация выгрузки работ.

Для работы с материалами курса отдельным аналогом можно считать **поиск Word**, но он не позволяет работать с единой базой лабораторных работ.

Также можно отметить использование **LLM-сервисов**, которые умеют искать по смыслу, но требуют постоянного обновления контекста материалов, могут допускать неточности и для регулярного использования часто требуют платного доступа.

---

Для backend части был выбран **Python**, так как он позволяет быстро разрабатывать микросервисную архитектуру и обладает списком библиотек.  
В качестве основного API-фреймворка был выбран **FastAPI** благодаря удобной валидации данных и хорошей поддержке REST API.  
Для клиентской части выбран **React**, поскольку он позволяет гибко строить интерфейс и удобно разделять приложение на независимые части.  
Для хранения данных выбраны:
- **PostgreSQL** — для реляционной части ядра системы;
- **MongoDB** — нереляционная база данных хранения материалов лабораторных работ;
Для полнотекстового поиска был выбран движок **Meilisearch** засчет простоты внедрения и встроенной поддержки поиска с опечатоками.

На слайде представлена диаграмма компонентов системы. 
На диаграмме выделены 3 уровня: клиентский, сервисный и хранилица системы.

---

Для демонстрации работы сервиса был разработана необходимая часть системы ядра для загрузки работ студентами.

Для нереляционной базы данных Wiki была разработана uml диаграмма. 

На диаграмме отражается корневой документ MongoLabDocument, соответствующий одной лабораторной работе, а также вложенные сущности LabSection и MaterialAsset. Связи между ними показывают, что один документ лабораторной работы содержит множество секций и одна секция множество изображений.


---

Для конвертации документов из Word была спроектирована бизнес-логика, представленная на диаграмме активностей.

Сначала сервис получает исходный документ и извлекает из него XML структуру. После этого пайплайн выделяет заголовки, абзацы, таблицы, изображения и другие части документа.

Далее система восстанавливает логическую структуру лабораторной работы, разбивает ее на секции и сохраняет их в виде отдельных markdown-файлов.

После завершения конвертации итоговый объект лабораторной работы загружается в MongoDB.

В ходе реализации была подготовлена общая инфраструктура проекта на основе Docker.

Перейдем к демонстрации работы сервиса:

Сервис запускается единой командой через Docker.
Перейдем на сайт и авторизируемся в системе.
После авторизации студент видит список доступных ему для сдачи работ и их статусы.
В хранилище оторажаются 4 работы, которые загрузил студент.
Вернемся на сайт и загрузим отчет и код для лабораторной работы. Так же можно добавить необязательный комментарий для проверяющих.
Статус работы в системе изменился и в хранилище появилась новая работа.

Перейдем к wiki части системы. Изначально материалы работ находятся в Word файлах, которые алгоритм конвертирует в удобное представление для web и мобильного интерфейса.
Алгоритм не просто считывет текст, а извлекает полную структуру документа с учетом списков, таблиц, формул и изображений.

В хранилище конвертированные материалы делятся на секции для удобного поиска по ним.

Вернемся на сайт и посмотрим эту же лабораторную работу. В итоговом виде корректно отображаются блоки кода, таблицы, формулы, изображения и списки.

Перейдем к системе поиска и попробуем найти теоритическую информацию в системе, например для коллекций. В поиске корректно отбразилась ссылка на теоритическую информацию по теме.
Так же попробуем найти теорию по теме Linq. Аналогично отобразилась секция с теоритическими материалами.

Теперь усложним поиск и добавим орфографичекие ошибки в запрос. Например не правильно напишем делигаты. Итогом так же являются теоритические материалы по теме.

Добавим фильтры, чтобы в поиске была только теория и другие секции не встречались.
