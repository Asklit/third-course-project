# Поиск В Wiki

## Последовательность действий

- Получить HTTP-запрос `GET /search` в `wiki/src/main.py`
- Прочитать параметры `q`, `tag`, `kind`, `lab_slug`, `limit`
- Попытаться выполнить поиск через `Meilisearch`
- Если `Meilisearch` недоступен, перейти на fallback-поиск по MongoDB и Python-эвристике
- Для `Meilisearch` использовать индекс секций `wiki_sections`
- Передать в `Meilisearch` текст запроса и фильтры
- Получить список секций из `Meilisearch`
- Преобразовать найденные документы к формату `SearchHit`
- Если `Meilisearch` не ответил, выбрать документы ЛР из MongoDB коллекции `labs`
- Для fallback-поиска пройти по секциям и проверить совпадение `_matches_query(...)`
- Для fallback-поиска вычислить score через `_score_hit(...)`
- Для fallback-поиска сформировать `snippet` через `_build_snippet(...)`
- Отсортировать результаты и вернуть `SearchResponse { total, items[] }`

## Где находится логика

Основная логика поиска находится в файлах:

- `wiki/src/main.py`
- `wiki/src/application/services/search_index.py`

Поиск вызывается через endpoint:

- `GET /search`

## Текущая архитектура поиска

Сейчас поиск устроен так:

- `wiki` хранит материалы в MongoDB
- `wiki` индексирует секции в `Meilisearch`
- `wiki` отдает наружу единый API `/search`
- `core` просто проксирует `/wiki/search`

То есть `Meilisearch` — внутренний search engine сервиса `wiki`, а не отдельный публичный API для клиента.

## Какие параметры поддерживаются

Поддерживаются параметры:

- `q` — текст запроса
- `tag` — фильтр по тегу
- `kind` — фильтр по типу секции
- `lab_slug` — искать внутри одной ЛР
- `limit` — ограничение числа результатов

## Что индексируется в Meilisearch

Один документ индекса = одна секция ЛР.

В индекс попадают поля:

- `id`
- `lab_id`
- `lab_slug`
- `lab_title`
- `section_id`
- `section_title`
- `kind`
- `order`
- `tags`
- `content_plain`

`content_plain` — это markdown секции после упрощенной очистки от служебной разметки.

## Какие возможности сейчас дает Meilisearch

- typo tolerance
- prefix matching
- более стабильное ранжирование
- фильтры по `lab_slug`, `kind`, `tags`
- сортировка по `lab_id`, `order`
- crop для фрагмента результата
- словарь синонимов для терминов вроде `ооп`, `oop`, `console`, `c#`

## Формула score fallback-поиска

Если `Meilisearch` недоступен, используется текущая эвристика score.

Обозначения:

- `Q` — нормализованный полный запрос
- `T(q)` — токены запроса
- `V(t)` — варианты токена `t` с учетом синонимов
- `ST` — заголовок секции
- `LT` — заголовок ЛР
- `TG` — объединенный список тегов ЛР и секции
- `TXT` — текст секции

Формула:

```text
score = S_full + S_tokens

S_full =
  120, если Q входит в ST
+  80, если Q входит в LT
+ 140, если Q точно совпадает с одним из TG
+  40, если Q входит в TXT

S_tokens = сумма по всем t из T(q) и всем v из V(t):
   40, если v входит в ST
 + 20, если v входит в LT
 + 50, если v точно совпадает с одним из TG
 + 10, если v входит в TXT
```

## Когда срабатывает fallback

Fallback используется, если:

- `Meilisearch` отключен
- `Meilisearch` не поднялся
- `Meilisearch` временно не отвечает

В этом случае `wiki` не падает и продолжает искать по MongoDB + Python-эвристике.

## Практический эффект после внедрения

Что было слабым у старого поиска:

- плохо работали неполные запросы
- плохо работали опечатки
- выдача была слишком зависима от ручной эвристики
- не было нормального search-engine поведения на росте материалов

Что дает новая схема:

- запросы типа `консол`, `коллекц`, `delegat` должны отрабатывать лучше
- короткие технические термины находят секции устойчивее
- качество поиска меньше зависит от ручных правил в коде
- архитектура готова к дальнейшему росту wiki
