## Идея

LMS строится как набор сервисов, где `core` является единой точкой входа для клиента.

Базовый поток:
1. Преподаватель/система открывает дедлайн и задание. (на других микросервисах)
2. Студент в `frontend` видит доступные задания и материалы.
3. Студент загружает решение (код + отчет), запрос уходит в `core`.
4. `core` сохраняет данные/метаданные, логирует событие, при необходимости дергает другие микросервисы.
5. Материалы ЛР запрашиваются через `core`, а фактически отдаются `wiki`.

## Зафиксированные решения (MVP)

- [x] Архитектура: 3 контейнера (`core`, `wiki`, `frontend`)
- [x] Единый API для клиента: все запросы через `core`
- [x] Авторизация: JWT
- [x] `core` БД: PostgreSQL
- [x] `wiki` хранилище: MongoDB (материалы ЛР в формате `.md`)
- [x] На фронтенде нужна реальная загрузка файлов и передача в `core`
- [x] Логи обязательны (демонстрация работы системы)
- [x] В MVP реализуем только роль `student` (без ролей `teacher/assistant/manager`)

## MVP Scope

### 1) Core (server / API gateway)

- [ ] Базовая структура FastAPI-проекта (`app`, `api`, `services`, `models`, `schemas`)
- [ ] Подключение PostgreSQL (SQLAlchemy + Alembic)
- [ ] JWT auth для `student` в формате, совместимом с общим core:
  - [ ] `POST /auth/login`
  - [ ] `POST /auth/refresh`
- [ ] API для фронтенда:
  - [ ] список доступных заданий студента
  - [ ] карточка задания
  - [ ] прием сдачи задания (метаданные + файл(ы)) + фиксация факта сдачи
  - [ ] статус/история сдачи
- [ ] Proxy/aggregation endpoint для wiki (core -> wiki)
- [ ] Системные логи (HTTP, ошибки, ключевые бизнес-события)
- [ ] Healthcheck endpoint (`/health`)

### 2) Wiki microservice

- [ ] Отдельный FastAPI-сервис
- [ ] Хранилище материалов ЛР в MongoDB
- [ ] Модель материала:
  - [ ] `lab_id` / `slug`
  - [ ] `title`
  - [ ] `content_md`
  - [ ] ссылки на prereq/зависимости (опционально для дерева)
- [ ] API:
  - [ ] получить список ЛР
  - [ ] получить материал конкретной ЛР
- [ ] Минимальная валидация и логи
- [ ] Healthcheck endpoint

### 3) Frontend

- [ ] Авторизация (JWT)
- [ ] Страница/форма со списком доступных заданий
- [ ] Страница задания:
  - [ ] просмотр материалов (через `core`)
  - [ ] загрузка файлов (код, `doc`/`docx`)
  - [ ] отправка сдачи в `core`
- [ ] Отображение статуса сдачи
- [ ] Базовая обработка ошибок API

### 4) Docker

- [ ] `docker-compose` для `core`, `wiki`, `frontend`, `postgres`, `mongo`
- [ ] Внутренняя сеть между контейнерами
- [ ] Переменные окружения для URL/секретов/подключений к БД
- [ ] Volume:
  - [ ] для postgres
  - [ ] для файлов сдач (если локальное файловое хранение в MVP)
- [ ] Команда быстрого старта проекта

### Базовый вариант (рекомендуется для MVP)
- [ ] Хранить файлы локально в volume (S3 не обязателен в MVP)
- [ ] В БД хранить метаданные: путь, тип, размер, кто и когда загрузил
- [ ] Выделить абстракцию `storage` в core, чтобы потом легко заменить на S3/MinIO
- [ ] После приема файла `core` отправляет событие/запрос о новой сдаче в другие сервисы (интеграционный контракт)
- [ ] Формат API загрузки: `multipart/form-data`
  - [x] поле `files[]` для одного или нескольких файлов
  - [x] поле `submission_meta` (JSON-строка: `assignment_id`, `comment`, `submitted_at`)

### Enhanced (позже)
- [ ] Перейти на S3-совместимое хранилище (например, MinIO) — не делаем в MVP
- [ ] Presigned URLs / отдельный file-service (при масштабировании)

## Пояснения терминов

- **`multipart/form-data`**: формат HTTP-запроса для отправки файлов и обычных полей в одном запросе.
- **`files[] + submission_meta`**:
  - `files[]` — сами файлы (код, `doc/docx`);
  - `submission_meta` — строка JSON с метаданными сдачи (какая работа, какой студент, комментарий).
- **S3-совместимое хранилище**: хранилище файлов с API как у Amazon S3. Пример: `MinIO` (можно поднять локально в Docker).
- **Presigned URL**: временная ссылка, по которой клиент загружает файл напрямую в хранилище без проксирования файла через backend.
- **Отдельный file-service**: выделенный микросервис только для загрузки/скачивания/хранения файлов. Для MVP не нужен.
